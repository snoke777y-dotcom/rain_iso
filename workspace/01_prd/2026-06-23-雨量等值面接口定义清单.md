# 雨量等值面接口定义清单

## 1. 文档目标

本文用于固化雨量等值面浏览器端方案的接口契约，避免前后端、静态资产制作、Worker 计算和渲染联调过程中反复返工。

本文定义四类接口：

1. `静态空间资产接口`
2. `动态观测原始接口序列`
3. `Worker 消息协议`
4. `FrameResult 结果对象接口`

本文是以下文档的补充：

- 设计说明：`workspace/01_prd/2026-06-23-雨量等值面浏览器端详细设计说明.md`
- 开发任务清单：`workspace/02_tasks/2026-06-23-雨量等值面开发任务清单.md`
- 原始接口正式字段规范：`workspace/01_prd/2026-06-24-雨量等值面原始接口正式字段规范.md`

---

## 2. 设计原则

- 下载完成后再计算，因此动态数据必须可一次性完整校验
- 浏览器端以 TypedArray 为主要计算结构，接口设计优先考虑低拷贝和顺序内存
- 逻辑结构和物理组织分开定义
- 可以扩展字段，但不得随意变更既有字段语义
- 版本兼容必须显式声明，不允许隐式猜测

---

## 3. 统一约定

### 3.1 命名风格

- JSON 字段统一使用 `snake_case`
- TypeScript 内部可映射为 `camelCase`
- 布尔值字段使用 `is_`、`has_`、`allow_` 前缀

### 3.2 时间格式

- 统一使用 `ISO 8601` 字符串
- 明确包含时区偏移
- 示例：`2026-06-23T08:15:00+08:00`

### 3.3 坐标系约定

- 静态资产必须在 `asset_manifest` 中声明坐标系
- 网格计算必须使用`米制投影坐标系`
- 行政边界渲染数据可同时保留地图展示坐标系，但必须声明

必填字段：

- `grid_crs`
- `render_crs`

### 3.4 数值类型约定

- 网格索引：`Int32`
- 行列号：`Int32`
- 站点 ID：`String` 或 `Int32`
- 雨量值：`Float32`
- 掩膜：`Uint8`
- 质量标记：`Uint8`

### 3.5 缺省值约定

- 缺测值不允许混入数值数组
- 动态包原始雨量矩阵中，缺测值统一用 `NaN`
- 最终渲染结果中的无值网格统一用 `NaN`
- `rain_mask = 0` 表示不渲染

### 3.6 版本约定

定义三类版本：

- `asset_version`
- `protocol_version`
- `algorithm_profile_version`

说明：

- `asset_version`：静态空间资产版本
- `protocol_version`：接口协议版本
- `algorithm_profile_version`：算法参数集版本

## 4. 静态空间资产接口

### 4.1 目标

静态空间资产用于支持前端本地计算，运行时只读，不随任务变化。

### 4.2 推荐物理组织

推荐使用一个版本化资产目录或一个压缩包，包含：

- `asset_manifest.json`
- `grid_meta.bin`
- `grid_mask.bin`
- `grid_neighbors.bin`
- `station_to_grid.bin`
- `station_meta.json`
- `render_boundary.geojson`

如需进一步压缩，可使用：

- 单 zip 包
- 或 `manifest + 二进制分片`

但逻辑字段必须保持一致。

### 4.3 资产清单 `asset_manifest.json`

#### 逻辑结构

```json
{
  "protocol_version": "1.0.0",
  "asset_version": "2026-06-bj-grid-v1",
  "algorithm_profile_version": "rain-iso-profile-v1",
  "city_code": "beijing",
  "grid_resolution_m": 1000,
  "grid_rows": 0,
  "grid_cols": 0,
  "grid_count": 0,
  "valid_grid_count": 0,
  "grid_crs": "EPSG:xxxx",
  "render_crs": "EPSG:4326",
  "bbox_projected": [0, 0, 0, 0],
  "bbox_render": [0, 0, 0, 0],
  "files": {
    "grid_meta": "grid_meta.bin",
    "grid_mask": "grid_mask.bin",
    "grid_neighbors": "grid_neighbors.bin",
    "station_to_grid": "station_to_grid.bin",
    "station_meta": "station_meta.json",
    "render_boundary": "render_boundary.geojson"
  },
  "checksums": {
    "grid_meta": "sha256:...",
    "grid_mask": "sha256:...",
    "grid_neighbors": "sha256:...",
    "station_to_grid": "sha256:...",
    "station_meta": "sha256:...",
    "render_boundary": "sha256:..."
  }
}
```

#### 必填字段

- `protocol_version`
- `asset_version`
- `grid_resolution_m`
- `grid_rows`
- `grid_cols`
- `grid_count`
- `grid_crs`
- `render_crs`
- `files`
- `checksums`

### 4.4 `grid_meta.bin`

#### 目标

定义每个网格的固定拓扑与空间定位。

#### 推荐逻辑字段

每个网格记录包含：

- `grid_id: Int32`
- `row: Int32`
- `col: Int32`
- `center_x: Float32`
- `center_y: Float32`

可选扩展：

- `min_x: Float32`
- `min_y: Float32`
- `max_x: Float32`
- `max_y: Float32`

#### 推荐组织方式

采用“字段分列”的 TypedArray 结构，而不是对象数组：

- `grid_id[]`
- `row[]`
- `col[]`
- `center_x[]`
- `center_y[]`

原因：

- 更适合浏览器顺序访问
- 更适合 Worker 与 GPU 上传

### 4.5 `grid_mask.bin`

#### 目标

标记网格是否属于北京市有效计算区域。

#### 逻辑结构

- 长度必须等于 `grid_count`
- 类型为 `Uint8Array`
- 值域：
  - `0`：无效区域
  - `1`：有效区域

### 4.6 `grid_neighbors.bin`

#### 目标

提供八邻域拓扑索引，避免运行时临时求邻居。

#### 逻辑结构

每个网格固定 8 个邻居槽位：

- `n`
- `ne`
- `e`
- `se`
- `s`
- `sw`
- `w`
- `nw`

推荐组织：

- `Int32Array[grid_count * 8]`

取值规则：

- 有邻居：填邻居 `grid_id`
- 无邻居：填 `-1`

### 4.7 `station_to_grid.bin`

#### 目标

提供站点到固定网格的直接映射。

#### 逻辑结构

每个站点至少包含：

- `station_id`
- `grid_id`

推荐组织方式：

- 若 `station_id` 为整数，可用二进制并行数组
- 若 `station_id` 为字符串，则建议：
  - `station_meta.json` 中保留站点顺序
  - `station_to_grid.bin` 仅保存与顺序对齐的 `grid_id[]`

推荐字段：

- `station_index: Int32`
- `grid_id: Int32`

### 4.8 `station_meta.json`

#### 目标

保存站点静态元信息与空间映射辅助字段。

#### 逻辑结构

```json
{
  "station_count": 0,
  "stations": [
    {
      "station_id": "string-or-number",
      "station_name": "string",
      "lon": 0,
      "lat": 0,
      "x": 0,
      "y": 0,
      "is_fortress_anchor": false,
      "is_tongzhou_anchor": false,
      "is_cross_boundary_anchor": false
    }
  ]
}
```

#### 必填字段

- `station_id`
- `lon`
- `lat`
- `is_fortress_anchor`
- `is_tongzhou_anchor`
- `is_cross_boundary_anchor`

#### 当前实现说明

当前代码实现中，固定锚点主数据已切换为外部字典文件：

- `datas/03_dictionary/rain_iso/fixed_anchor_stations.json`

运行时流程为：

1. 读取固定锚点字典
2. 读取 `station_meta.json`
3. 以 `station_id` 求交集，生成当前资产范围内的 `fixed_anchor_station_ids`

说明：

- `station_meta.json` 中的三类布尔字段仍可保留，作为兼容字段或后续资产生产字段
- 但不再作为当前实现的唯一固定锚点来源

### 4.9 `render_boundary.geojson`

#### 目标

用于地图展示阶段的边界绘制和裁切参考。

#### 要求

- 必须与 `render_crs` 一致
- 可为 `Polygon` 或 `MultiPolygon`
- 仅服务渲染，不参与核心数值计算

### 4.10 静态资产校验规则

前端加载资产时必须校验：

1. `asset_version` 存在
2. `grid_rows * grid_cols >= grid_count`
3. `grid_mask.length == grid_count`
4. `grid_neighbors.length == grid_count * 8`
5. `station_meta.station_count == station_to_grid.length`
6. 文件 checksum 一致

任一失败则：

- 禁止启动计算
- 返回 `ASSET_VALIDATION_FAILED`

## 5. 动态观测数据接口

### 5.1 目标

动态观测输入改为两份原始接口返回文件，分别提供：

1. `5分钟分时序列`
2. `1小时累计序列`

前端不再从 `5分钟` 基础数据推导累计产品，而是直接对两份接口返回做排序、对齐与拆帧。

### 5.2 推荐物理组织

当前推荐直接落地两份原始 JSON 文件：

- `realtime_5m_response.json`
- `realtime_1h_response.json`

当前已确认：

- 本地样例文件即真实接口返回数据
- 后续接口协议按样例字段结构冻结
- 正式文档以本节定义为准

当前样例源文件位于：

- `datas/01_raw/realtime_5m_response.txt`
- `datas/01_raw/realtime_1h_response.json.txt`

两份文件顶层结构保持一致：

```json
{
  "data": {
    "2026-06-24 13:55:00": [
      {
        "stcd": "A001",
        "stnm": "站点名称",
        "lgtd": 116.1,
        "lttd": 39.9,
        "drp": 0.2
      }
    ]
  },
  "msg": "操作成功",
  "code": "0"
}
```

### 5.3 顶层字段约定

顶层字段集合按样例冻结为：

- `data`
- `msg`
- `code`

- `code`
  - `"0"` 表示接口成功
- `msg`
  - 接口说明文字
- `data`
  - `时间字符串 -> 站点数组`

说明：

- 时间键当前样例中并不保证天然有序
- 同一时间片内站点数组顺序也不保证稳定
- 前端必须按时间升序和 `stcd` 稳定站点序重建内部矩阵

### 5.4 单站记录字段

单站记录字段按样例冻结为以下完整集合：

- `sysid: string`
- `stcd: string`
- `stnm: string`
- `rvnm: string`
- `hnnm: string`
- `lgtd: number`
- `lttd: number`
- `stlc: string`
- `addvcd: string`
- `addvnm: string`
- `adnm: string`
- `stlvl: string`
- `admauth: string`
- `isOut: string`
- `drp: number`
- `bscd: string`
- `bsnm: string`
- `area: string`
- `star: number`

当前前端计算直接依赖的核心字段为：

- `stcd`
- `stnm`
- `lgtd`
- `lttd`
- `drp`

其余字段当前作为透传元信息保留，用于：

- 后续站点分类
- 调试与核对
- 辅助质量分析

### 5.5 前端内部整理规则

前端加载两份原始接口后，统一整理为内部直接序列：

1. 取出全部时间键
2. 转为 `ISO 8601 +08:00`
3. 对时间升序排序
4. 以全部 `stcd` 建立稳定站点顺序
5. 转为 row-major 的 `Float32Array`

说明：

- `5分钟分时` 与 `1小时累计` 都走同一套整理逻辑
- 差别仅在于时间步长和产品类型不同
- 当前样例中 `5分钟` 有 `2` 个时间片、`1小时` 有 `10` 个时间片，二者首帧站点数均为 `4137`
- 上述帧数仅代表当前样例，不属于协议常量；站点字段结构才是冻结部分

### 5.6 动态输入校验规则

前端加载原始接口文件时必须校验：

1. `code == "0"`
2. `data` 非空
3. 每个时间片都至少包含一个站点
4. 同一产品内各时间片站点集合一致
5. `5分钟分时` 序列相邻时间差为 `5分钟`
6. `1小时累计` 序列相邻时间差为 `60分钟`
7. 两份接口整理后的 `station_ids` 集合一致

任一失败则：

- 禁止启动计算
- 返回 `PACKAGE_VALIDATION_FAILED`

## 6. Worker 消息协议

### 6.1 目标

主线程与 Worker 之间通过结构化消息通信，消息体必须可序列化，数值数组优先通过 `Transferable` 传输。

### 6.2 主线程 -> Worker

#### 6.2.1 `load_assets`

用于加载静态资产。

```json
{
  "type": "load_assets",
  "request_id": "string",
  "payload": {
    "asset_manifest": {},
    "grid_meta": {},
    "grid_mask": {},
    "grid_neighbors": {},
    "station_to_grid": {},
    "station_meta": {}
  }
}
```

说明：

- 二进制数组通过 `Transferable` 传入

#### 6.2.2 `detect_backend`

用于探测本次会话可用后端。

```json
{
  "type": "detect_backend",
  "request_id": "string",
  "payload": {}
}
```

#### 6.2.3 `start_task`

用于启动整包计算任务。

```json
{
  "type": "start_task",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "rain_5m_sequence": {},
    "accum_1h_sequence": {},
    "preferred_backend": "auto",
    "algorithm_profile_version": "rain-iso-profile-v1"
  }
}
```

`preferred_backend` 取值：

- `auto`
- `webgpu`
- `webgl2`
- `cpu`

说明：

- `rain_5m_sequence` 为前端整理后的直接序列
- `accum_1h_sequence` 为前端整理后的直接序列
- 正式规则仍是“先探测再决定”
- `preferred_backend` 只用于调试或强制覆盖

#### 6.2.4 `cancel_task`

```json
{
  "type": "cancel_task",
  "request_id": "string",
  "payload": {
    "task_id": "string"
  }
}
```

#### 6.2.5 `release_frame_cache`

```json
{
  "type": "release_frame_cache",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "frame_keys": ["string"]
  }
}
```

### 6.3 Worker -> 主线程

#### 6.3.1 `assets_loaded`

```json
{
  "type": "assets_loaded",
  "request_id": "string",
  "payload": {
    "asset_version": "string",
    "grid_count": 0,
    "station_count": 0
  }
}
```

#### 6.3.2 `backend_detected`

```json
{
  "type": "backend_detected",
  "request_id": "string",
  "payload": {
    "selected_backend": "webgpu",
    "available_backends": ["webgpu", "cpu"]
  }
}
```

#### 6.3.3 `task_started`

```json
{
  "type": "task_started",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "selected_backend": "webgpu",
    "total_frames": 0
  }
}
```

说明：

- `selected_backend` 表示任务启动时选中的后端
- 若运行中发生后端降级，后续单帧结果以内层 `frame_result.selected_backend` 的实际值为准

#### 6.3.4 `task_progress`

```json
{
  "type": "task_progress",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "completed_frames": 0,
    "total_frames": 0,
    "current_frame_key": "string",
    "phase": "propagating"
  }
}
```

`phase` 建议取值：

- `validating`
- `preprocessing`
- `selecting_anchors`
- `seeding_grid`
- `building_mask`
- `propagating`
- `smoothing`
- `assembling`

#### 6.3.5 `frame_ready`

```json
{
  "type": "frame_ready",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "frame_key": "string",
    "frame_result": {}
  }
}
```

#### 6.3.6 `task_completed`

```json
{
  "type": "task_completed",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "completed_frames": 0,
    "total_frames": 0,
    "elapsed_ms": 0
  }
}
```

#### 6.3.7 `task_failed`

```json
{
  "type": "task_failed",
  "request_id": "string",
  "payload": {
    "task_id": "string",
    "error_code": "PACKAGE_VALIDATION_FAILED",
    "message": "string",
    "details": {}
  }
}
```

#### 6.3.8 `task_cancelled`

```json
{
  "type": "task_cancelled",
  "request_id": "string",
  "payload": {
    "task_id": "string"
  }
}
```

### 6.4 错误码建议

- `ASSET_VALIDATION_FAILED`
- `PACKAGE_VALIDATION_FAILED`
- `BACKEND_UNAVAILABLE`
- `TASK_ALREADY_RUNNING`
- `TASK_NOT_FOUND`
- `GPU_CONTEXT_LOST`
- `GPU_MEMORY_EXCEEDED`
- `FRAME_COMPUTE_FAILED`
- `UNKNOWN_ERROR`

## 7. `FrameResult` 结果对象接口

### 7.1 目标

`FrameResult` 是单帧最终产物，既服务渲染，也服务网格值查询和回放切换。

### 7.2 逻辑结构

```json
{
  "frame_key": "rain_5m|2026-06-23T08:15:00+08:00",
  "frame_type": "rain_5m",
  "frame_time": "2026-06-23T08:15:00+08:00",
  "selected_backend": "webgpu",
  "legend_id": "legend_5m_v1",
  "value_grid": "transferable",
  "rain_mask": "transferable",
  "hard_anchor_mask": "transferable",
  "soft_obs_mask": "transferable",
  "summary": {
    "max_value": 0,
    "renderable_grid_count": 0,
    "hard_anchor_count": 0,
    "soft_obs_count": 0,
    "suspect_station_count": 0,
    "ordinary_only_mode": false
  }
}
```

说明：

- `selected_backend` 表示该帧实际执行后端
- 当任务启动时选择 `webgl2`，若运行中回退到 CPU，则该字段可为 `cpu`

### 7.3 字段说明

#### `frame_key`

建议规则：

`{frame_type}|{frame_time}`

示例：

- `rain_5m|2026-06-23T08:15:00+08:00`
- `accum_1h_step|2026-06-23T09:15:00+08:00`

#### `frame_type`

取值：

- `rain_5m`
- `accum_1h_step`

#### `value_grid`

- 类型：`Float32Array`
- 长度：`grid_count`
- 有值网格：`>= 0.1`
- 不显示网格：`NaN`

#### `rain_mask`

- 类型：`Uint8Array`
- 长度：`grid_count`
- `0`：不渲染
- `1`：渲染

#### `hard_anchor_mask`

- 类型：`Uint8Array`
- 长度：`grid_count`
- `1` 表示该格为硬锚点格

#### `soft_obs_mask`

- 类型：`Uint8Array`
- 长度：`grid_count`
- `1` 表示该格由普通站直接落格形成

### 7.4 `summary` 字段

必填字段：

- `max_value`
- `renderable_grid_count`
- `hard_anchor_count`
- `soft_obs_count`
- `suspect_station_count`
- `ordinary_only_mode`

可选扩展：

- `min_renderable_value`
- `mean_renderable_value`
- `elapsed_ms`

### 7.5 渲染侧最小入参

页面渲染连续色带面，最小需要：

- `frame_type`
- `frame_time`
- `legend_id`
- `value_grid`
- `rain_mask`

网格值查询最小需要：

- `value_grid`
- `grid_meta`

### 7.6 结果一致性要求

对同一帧：

- `value_grid.length == grid_count`
- `rain_mask.length == grid_count`
- `hard_anchor_mask.length == grid_count`
- `soft_obs_mask.length == grid_count`
- 任一 `hard_anchor_mask[i] == 1` 时，不允许 `value_grid[i]` 为 `NaN`

## 8. 图例接口

### 8.1 目标

图例本身是固定配置，但主线程渲染仍需要统一获取接口。

### 8.2 推荐逻辑结构

```json
{
  "legend_id": "legend_5m_v1",
  "product_type": "rain_5m",
  "bins": [
    { "min": 0.1, "max": 0.3, "color": "#xxxxxx" },
    { "min": 0.4, "max": 0.9, "color": "#xxxxxx" }
  ]
}
```

### 8.3 固定图例 ID

建议固定：

- `legend_5m_v1`
- `legend_accum_24h_v1`

## 9. 参数配置接口

### 9.1 目标

将可调算法参数和结构字段分离，避免把参数硬编码到协议主体里。

### 9.2 推荐结构

```json
{
  "algorithm_profile_version": "rain-iso-profile-v1",
  "params": {
    "render_threshold_mm": 0.1,
    "rain_mask_radius_min_5m": 3,
    "rain_mask_radius_max_5m": 10,
    "rain_mask_radius_min_accum": 6,
    "rain_mask_radius_max_accum": 20,
    "propagation_weight_neighbor": 0.7,
    "propagation_weight_anchor": 0.3,
    "smooth_rounds_5m": 2,
    "smooth_rounds_accum": 4,
    "soft_obs_delta_limit_mm": 1.0
  }
}
```

说明：

- 可由主线程随 `start_task` 一并传入
- 若未传入，则 Worker 使用默认 profile

## 10. 接口校验顺序

建议主线程和 Worker 按以下顺序校验：

1. 校验静态资产
2. 校验动态数据包
3. 校验 `asset_version` 一致性
4. 校验 `protocol_version` 兼容性
5. 校验算法参数 profile
6. 校验后端可用性
7. 启动任务

说明：

- 未通过任一步，不得进入计算阶段

---

## 11. 最小联调清单

在正式开发并行前，至少要完成以下联调样例：

1. `静态资产可加载`
2. `动态包可校验`
3. `start_task -> frame_ready -> task_completed` 全链路可跑通
4. 分时产品可返回 `FrameResult`
5. 累计产品可返回 `FrameResult`
6. 图例 ID 与产品类型绑定正确

---

## 12. 接口冻结建议

以下字段在首版联调前建议冻结，不再随意改名：

- `asset_version`
- `protocol_version`
- `algorithm_profile_version`
- `frame_type`
- `frame_time`
- `value_grid`
- `rain_mask`
- `hard_anchor_mask`
- `soft_obs_mask`
- `ordinary_only_mode`

原因：

- 这些字段贯穿资产、计算、渲染和回归测试
- 改名会引发整链路返工

---

## 13. 结论

本清单已经把雨量等值面方案最关键的接口契约定死为四类：

- 静态空间资产
- 动态观测原始接口序列
- Worker 消息协议
- `FrameResult` 结果对象

按本文接口推进，可以直接支持后续：

- 前后端并行联调
- Worker 计算链路开发
- GPU / CPU 双后端接入
- 页面渲染与时间切换实现
