# 雨量等值面浏览器端详细设计说明

## 1. 文档目标

本文用于固化“浏览器端生成雨量等值面”的最新方案，作为直接进入开发的设计基线。

本文覆盖：

- 产品定义与时间语义
- 站点参与规则与锚点规则
- 输入输出与数据契约
- 前端计算架构
- 单帧算法链路
- GPU / CPU 双后端执行策略
- 图例、渲染和无雨区规则
- 性能目标、并发策略、验收标准

本文不覆盖：

- 服务端数据库设计
- 后台站点配置维护界面
- 独立等值线矢量产品

## 2. 已确认业务规则

### 2.1 产品类型

系统生成两类产品：

1. `5分钟分时雨量等值面`
2. `起报累计雨量等值面`

### 2.2 时间语义

#### 5分钟分时雨量等值面

- 基础时间分辨率为 `5分钟`
- 每个 5 分钟时刻生成一张分时等值面
- 该帧仅表示当前 5 分钟降雨量

#### 起报累计雨量等值面

- 累计产品的输出步长为 `1小时`
- 累计语义为 `起报累计`
- 累计值由服务端接口直接提供
- 前端按接口返回的每个 `1小时` 时刻直接生成一张累计等值面

说明：

- 不采用“最近 1 小时滚动累计”
- 前端不再从 `5分钟` 序列推导累计产品

### 2.3 网格与空间范围

- 全市采用预生成的 `1km x 1km` 规则网格
- 网格不在运行时生成
- 使用北京市行政边界 `mask` 裁切有效区域
- 运行时仅在有效网格内计算

### 2.4 计算位置与任务触发

- 计算必须在浏览器客户端完成
- 不占用服务器计算资源
- 服务端仅负责提供数据包和静态索引资产
- 用户点击后，必须先完整下载当前时间范围的数据包
- 下载完成并校验通过后，才开始本地批量计算
- 不采用边下载边计算

### 2.5 参与计算的站点集合

每个时间帧都先读取`全部站点`，但并非全部原样进入成图。

当前帧的有效站点定义为：

- 非缺测
- 非负值
- 非非法值
- 通过基础质量校验
- `P > 0`

说明：

- 所有 `P = 0` 的站点不参与该帧计算
- 所有有效站点都参与等值面计算
- 参与方式分为 `硬锚点`、`软观测点`、`待补全格`

### 2.6 锚点定义

锚点分为两类。

#### 固定锚点

- `121个水文重点堡垒站`
- `5个通州雨量站`
- `跨界沟道站`

#### 动态锚点

- `当前帧 Top30 站点`

Top30 口径：

- 分时产品：按当前 `5分钟` 雨量值排序取前 30
- 累计产品：按当前累计帧的 `起报累计值` 排序取前 30

说明：

- 锚点总数不是固定 `126`
- 当前设计为 `固定锚点 + 动态锚点`
- Top30 与固定锚点按站点 ID 去重

### 2.7 站点层级定义

单帧计算中的站点与网格约束分为三层：

1. `硬锚点`
2. `软观测点`
3. `待补全格`

说明：

- 硬锚点：固定锚点和动态锚点，值不可被平滑改写
- 软观测点：全部有效站点中，除锚点外直接落格形成的观测格，当前口径下不允许低于原始观测值，只允许按上限阈值小幅上浮
- 待补全格：既无锚点也无普通站落格的网格，完全依赖传播和补全生成

### 2.8 同网格站点冲突规则

#### 情况 A：同一网格中存在锚点

若同一网格中存在任意锚点，则该网格由锚点控制，普通站不再生效。

当前降级实现中，锚点冲突优先级顺序固定为：

1. `全部固定锚点`
2. `当前帧 Top30 动态锚点`

同优先级内取最大值。

说明：

- 固定锚点在前端运行时统一视为同一优先级
- 不再对 `重点堡垒站 / 通州5站 / 跨界沟道站` 做分类分层处理

#### 情况 B：同一网格中没有锚点，但有多个普通站

若同一网格中没有锚点但有多个普通站：

1. 先找到该网格最近的`活动锚点网格`
2. 取该活动锚点网格的值作为参考值
3. 在多个普通站中，选择`数值最接近该参考值`的那个站值作为该网格初值

若多个普通站与参考值距离相同，则按以下稳定顺序裁决：

1. 先取值更大的站
2. 若值也相同，则按 `stationId` 字典序取第一条

若本帧没有任何活动锚点，则采用回退规则：

- 同格多个普通站取最大值
- 结果标记为 `ordinary_only_mode`

### 2.9 连续传播规则

对仍然无值的网格，不采用“单纯取最大值”的旧规则，而采用`连续性优先的传播规则`。

目标：

- 让锚点控制主雨核
- 让普通站参与全域信息补充
- 让新补出的网格值尽量接近相邻已确定网格值

传播规则核心要求：

1. 先由已知网格场向外扩展
2. 网格甲被确定后，其周边待补全网格优先取`最接近网格甲`的数值
3. 若有多个候选来源，则优先选择使局部场连续的来源
4. 同时仍受最近活动锚点的参考值约束

推荐裁决分数：

`score = 0.7 * |v_candidate - v_neighbor_mean| + 0.3 * |v_candidate - v_anchor_ref|`

其中：

- `v_candidate`：候选来源网格值
- `v_neighbor_mean`：当前待补全格周边已确定网格的均值或主导值
- `v_anchor_ref`：最近活动锚点网格值

分数最小的候选值优先。

若本帧没有活动锚点，则仅采用连续性项：

`score = |v_candidate - v_neighbor_mean|`

### 2.10 平滑约束

- 先构建“部分已知格场”
- 再做连续传播补全
- 最后做受约束平滑

平滑约束如下：

- `硬锚点格` 值必须严格保持原值不变
- `软观测格` 不允许低于原始观测值，只允许在 `softObsMaxDelta` 上限内向上微调
- `待补全格` 允许自由度最大

### 2.11 无雨区与噪声规则

- 非雨区直接不渲染，视为 `null / transparent`
- `0 < P < 0.1mm` 统一视为噪声，不渲染
- `0.1mm` 为全局最小显示阈值
- 该规则同时适用于 `5分钟分时` 和 `起报累计` 两类产品

- 保证不同时间帧之间可比

## 4. 输入输出设计

### 4.1 输入数据

前端任务输入由两部分组成。

#### 4.1.1 静态空间资产

静态空间资产在项目初始化后长期复用，不随时间范围变化：

- `grid_meta`
  - 网格唯一 ID
  - 行列号
  - 网格中心点坐标
  - 网格边界框
- `grid_mask`
  - 是否位于北京市有效范围内
- `grid_neighbors`
  - 八邻域索引
- `station_to_grid`
  - 站点 ID 到网格 ID 的映射
- `station_meta`
  - 站点坐标
  - 站点类别
  - 可选保留固定锚点分类字段
  - 当前运行时固定锚点主数据来源于外部字典文件与 `station_meta` 的站点交集
- `render_boundary`
  - 地图渲染所需边界数据

#### 4.1.2 动态观测原始接口序列

动态数据按用户选择的时间范围一次性下载。

当前口径要求服务端直接下发两份原始接口文件：

- `5分钟分时序列`
- `1小时累计序列`

说明：

- 本地样例文件即真实接口返回
- 前端后续按该字段结构冻结实现

推荐字段：

- `task_id`
- `asset_version`
- `time_range`
  - `start_time`
  - `end_time`
- `realtime_5m_response`
- `realtime_1h_response`

说明：

- 前端只做加载、校验、排序、透传和拆帧
- 前端不再从 `5分钟` 数据推导累计产品
- 两类产品均以接口原始返回为准

### 4.2 输出结果

每个时间帧输出一个 `FrameResult`。

推荐字段：

- `frame_type`
  - `rain_5m`
  - `accum_1h_step`
- `frame_time`
- `value_grid`
  - 最终网格雨量值数组
- `rain_mask`
  - 可渲染网格掩膜
- `hard_anchor_mask`
  - 硬锚点格掩膜
- `soft_obs_mask`
  - 软观测格掩膜
- `anchor_grid`
  - 锚点网格位置和锚点值
- `legend_bins`
  - 当前产品绑定的固定图例
- `summary`
  - 最大值
  - 非空网格数量
  - 活动锚点数
  - 软观测格数量
  - 疑似异常站点数
  - 是否处于 `ordinary_only_mode`
- `backend`
  - `webgpu`
  - `webgl2`
  - `cpu`

## 6. 计算流程

### 6.1 总流程

1. 用户选择时间范围
2. 主线程请求下载动态原始接口数据
3. 下载完成后执行完整性校验
4. Worker 探测可用后端
5. 构建 5 分钟帧序列和累计帧序列
6. 按时间顺序逐帧计算
7. 每完成一帧即产出 `FrameResult`
8. 主线程缓存并渲染当前帧
9. 用户按时间顺序切换查看

### 6.2 单帧流程

1. 读取当前帧全部站点值
2. 执行硬过滤和软异常判定
3. 排除 `P = 0` 和无效站点
4. 识别固定锚点和动态锚点
5. 生成普通有效站集合
6. 全部有效站点直接落格
7. 先确定硬锚点格
8. 再确定普通观测格
9. 构建部分已知格场
10. 生成本帧 `rain_mask`
11. 在 `rain_mask` 内执行连续传播
12. 在 `rain_mask` 内执行受约束平滑
13. 应用最小显示阈值 `0.1mm`
14. 输出最终 `FrameResult`

## 7. 单帧算法详细设计

### 7.1 站点预处理

每帧站点先做基础清洗。

#### 硬过滤

以下情况直接剔除：

- 缺测
- 非数值
- 负值
- 站点 ID 不存在于静态映射表

#### 软异常判定

异常值按数值与空间邻域共同判定，但不使用 `admauth / stlvl / star / isOut` 参与规则。

异常状态建议分为：

- `normal`
- `suspect`
- `invalid`

当前冻结规则：

1. 先执行硬无效判定：
   - 非数值 -> `invalid`
   - 非有限值 -> `invalid`
   - 负值 -> `invalid`
2. `P = 0` 不进入异常值判定，直接按零值过滤排除
3. 对其余正值站点，取站点周边 `5km` 邻域作为参考范围
4. 邻域参考站排除待判定站自身
5. `5km` 邻域参考站数量少于 `4` 个时，使用静态 `fallback_nearest_neighbors` 补足参考站
6. 若补足后参考站仍少于 `4` 个，则不启用空间异常判定，仅保留硬无效规则，其余按 `normal`
7. 取邻域最大值与第二大值，若 `最大值 <= 第二大值 * 2`，则认为最大值可信
8. 若最大值不可信，则回退为“仅保留硬无效规则，其余按 `normal`”
9. `量级 = 当前产品图例档位`
10. 以可信最大值所在档位为中心：
   - 中心档位及上下相邻各 `1` 档 -> `normal`
   - 再向外扩 `1` 档 -> `suspect`
   - 超出上述范围 -> `invalid`

状态作用：

- `normal`：正常参与计算，可参与动态锚点 Top30
- `suspect`：参与普通站落格，但默认不进入动态锚点 Top30
- `invalid`：直接剔除，不参与后续成图

#### 零值过滤

- 所有 `P = 0` 站点直接排除

### 7.2 锚点选择与站点分类

当前帧全部有效站点经过预处理后，按以下步骤分类：

1. 识别固定锚点站
2. 在剩余有效站点中计算当前帧 Top30
3. 生成动态锚点集合
4. 锚点去重
5. 其余有效站点归为普通站

输出三类集合：

- `hard_anchor_stations`
- `ordinary_stations`
- `excluded_stations`

### 7.3 直接落格与同格裁决

所有有效站点先通过 `station_to_grid` 直接映射到自身网格。

#### 子步骤 A：落锚点

- 先处理固定锚点
- 再处理动态锚点
- 同格锚点按“固定锚点统一优先，动态锚点次之；同级取最大值”规则裁决

#### 子步骤 B：落普通站

仅在当前网格`没有锚点`时，普通站才可能生效。

若同格只有一个普通站：

- 直接采用该站值

若同格有多个普通站：

1. 找最近活动锚点格
2. 取该锚点值作为参考值
3. 选取与该参考值最接近的普通站值

若本帧无活动锚点：

- 同格普通站取最大值
- `summary.ordinary_only_mode = true`

输出：

- `seed_value_grid`
- `hard_anchor_mask`
- `soft_obs_mask`
- `known_mask`

### 7.4 雨区掩膜生成

由于 `P = 0` 站点完全不参与计算，且非雨区应透明，因此每帧必须先生成有限的 `rain_mask`。

当前设计不再只依赖锚点种子，而是依赖：

- `hard_anchor_mask`
- `soft_obs_mask`

推荐规则：

1. 以所有已知格为种子
2. 对硬锚点种子赋予更高膨胀权重
3. 结合种子密度和活动锚点间距估算本帧扩展半径
4. 在八邻域上生成 `rain_mask`
5. 与行政边界 `grid_mask` 相交

推荐参数：

- 分时产品：
  - `r_min = 3格`
  - `r_max = 10格`
- 累计产品：
  - `r_min = 6格`
  - `r_max = 20格`

说明：

- 当前阶段不把半径写死在算法内部
- 推荐做成可配置项并由样例数据校准

### 7.5 连续传播补全

在 `rain_mask` 内，对 `known_mask = false` 的网格执行补全。

传播原则：

- 传播源来自已确定网格
- 新网格值优先与周边已知网格值连续
- 同时受最近活动锚点参考值轻约束

推荐实现步骤：

1. 维护按波次扩展的待补全边界
2. 收集当前待补全格的已确定八邻域
3. 计算 `v_neighbor_mean`
4. 计算最近活动锚点参考值 `v_anchor_ref`
5. 对邻域候选值计算连续性分数
6. 取分数最小者作为当前网格值

若本帧没有活动锚点：

- 只依据已确定邻域连续传播
- 结果标记 `ordinary_only_mode = true`

### 7.6 受约束平滑

连续传播之后，需要对结果场做平滑以消除块状边界。

平滑原则：

- 仅在 `rain_mask` 内平滑
- `hard_anchor_mask = true` 的格子严格锁定
- `soft_obs_mask = true` 的格子不允许低于原始观测值
- 普通补全格允许更大自由度

推荐参数：

- 邻域核：`3x3`
- 分时产品：`8~12轮`
- 累计产品：`8~12轮`

推荐约束：

- 硬锚点格：不变
- 软观测格：下限锁定原始观测值，上限限制在配置阈值内
- 普通补全格：按平滑结果更新

### 7.6.1 当前联调口径

当前仓库联调默认值：

- `rounds = 12`
- `softObsMaxDelta = 20`

当前实现目标不是让软观测点跟着邻域均值回落，而是：

- 局部范围内已有观测量级必须保住
- 峰值可以向周边传播，但不能在平滑阶段被先压低

### 7.6.2 栅格渲染保峰规则

渲染阶段当前采用栅格超采样，不单独输出矢量等值线。

当前口径：

- 导出时使用 `pixelScale = 4`
- 局部采样窗口按有效角点取最大量级，不再做均值混色
- 图面中的规则点状痕迹，本质上是超采样后落在原始网格中心的像素，不是额外观测点
- 当前默认实现不再展示原始网格中心像素：可渲染雨格中心跟随相邻非中心像素，不可渲染格中心保持透明
- 目的不是追求连续插值视觉，而是保证局部最大量级不在渲染阶段被稀释

### 7.7 最小显示阈值

平滑结束后，执行最终阈值裁切：

- `value < 0.1mm` 的网格设为 `null`
- 同步将对应 `rain_mask` 位置置为 `false`

---

## 8. 分时与累计序列组织方式

### 8.1 基础原则

- 服务端分别下发：
  - `5分钟分时` 原始接口序列
  - `1小时累计` 原始接口序列
- 前端不再从 `5分钟` 数据推导累计产品
- 前端仅负责把两份原始接口返回整理为内部直接序列

### 8.2 前端整理规则

对两份接口返回，前端统一执行：

1. 读取 `data` 字段
2. 取出全部时间键
3. 转为 `ISO 8601 +08:00`
4. 按时间升序排序
5. 按 `stcd` 重建稳定站点顺序
6. 转为 row-major 数值矩阵
7. 逐帧拆分为内部序列

说明：

- 原始接口字段结构已按样例文件冻结
- 正式字段定义以接口定义清单第 `5` 章为准

### 8.3 累计产品的动态锚点规则

累计帧的 Top30 按当前累计值排序，不按单小时值排序。

---

## 9. GPU / CPU 双后端设计

### 9.1 后端选择顺序

任务开始前按以下顺序探测：

1. `WebGPU`
2. `WebGL2 + 浮点纹理支持`
3. `CPU`

### 9.2 后端职责划分

#### CPU 负责

- 数据包校验
- 时间序列组织
- 全站预处理
- 动态锚点 Top30 选择
- 站点分类
- 同格冲突裁决
- 普通站“最接近最近锚点值”选择
- 结果对象组装

#### GPU 优先负责

- `rain_mask` 形态扩展
- 已知格场传播补全
- 受约束平滑
- 渲染前纹理预处理

说明：

- CPU 必须先有基线实现
- GPU 以后端加速方式接入
- 不能先做 GPU 再回头定义业务规则

### 9.3 一致性要求

GPU 与 CPU 的最终结果必须满足：

- 同一帧同一网格的数值误差 `<= 0.05mm`
- 硬锚点格值完全一致
- 软观测格约束行为一致
- 图例分档一致

### 9.4 中途异常处理

单任务执行过程中不自动切换后端。

若出现以下情况：

- GPU 上下文丢失
- 显存不足
- 关键扩展不可用

则：

1. 当前任务标记失败
2. 返回明确错误状态
3. 由用户重新触发任务

---

## 10. 并发、缓存与内存控制

### 10.1 并发原则

- 结果按时间顺序产出
- 当前帧优先
- 后续帧后台顺序预计算

### 10.2 GPU 模式并发

- 同时仅维持 `1` 条 GPU 主计算流水线
- 最多允许 `1~2` 帧处于 in-flight 状态

### 10.3 CPU 模式并发

- Worker 并发度 = `min(物理核心数 - 1, 6)`

### 10.4 缓存策略

#### 静态资产缓存

- 网格、边界、邻接、站点映射长期常驻内存

#### 结果帧缓存

- 仅缓存最终 `FrameResult`
- 不长期保存所有中间数组
- 建议使用 LRU 控制内存

### 10.5 内存控制

- 单帧中间缓冲复用
- 帧完成后立即释放中间数组
- 渲染完成后仅保留结果数组和必要摘要

## 11. 性能目标

以下为目标值，不包含网络下载时间。

### 11.1 单帧性能目标

#### GPU

- 首帧冷启动：`< 0.8s`
- 后续单帧稳定计算：`30ms ~ 120ms`
- 用户感知渲染完成：`< 180ms`

#### CPU

- 首帧冷启动：`< 1.5s`
- 后续单帧稳定计算：`150ms ~ 900ms`
- 用户感知渲染完成：`< 1.2s`

### 11.2 整批任务目标

以 `24小时` 范围为例：

- `5分钟分时`：288 帧
- `起报累计`：24 帧
- 合计：312 帧

目标量级：

- GPU：`10s ~ 45s`
- CPU：`1min ~ 5min`

说明：

- 新方案引入“全站直接落格 + 连续传播”，较旧方案更重
- 首版开发以规则正确和稳定性优先

---

## 12. 错误处理与边界场景

### 12.1 无有效站点

若某帧经过过滤后没有有效站点：

- 该帧 `rain_mask` 全空
- 页面不渲染等值面
- 返回空结果摘要

### 12.2 无活动锚点但有普通站

若某帧没有活动锚点但有普通站：

- 允许进入 `ordinary_only_mode`
- 普通站直接落格
- 传播仅依赖连续性规则
- 摘要中显式标记该模式

### 12.3 数据包校验失败

若下载后的数据包：

- 时间帧缺失
- checksum 不一致
- 静态资产版本不匹配

则：

- 不启动计算
- 提示重新下载

### 12.4 后端不可用

若 GPU 能力检测失败：

- 直接切换到下一优先级后端
- 在任务启动前完成，不属于中途切换

---

## 13. 验收标准

### 13.1 功能验收

- 能生成 `5分钟分时雨量等值面`
- 能生成 `起报累计雨量等值面`
- 两类产品都能输出网格雨量值
- 页面按时间顺序逐张切换查看
- 图例固定，不因帧变化而变化
- `P = 0` 站点不参与计算
- `< 0.1mm` 不渲染
- 非雨区透明

### 13.2 规则验收

- 固定锚点与动态锚点识别正确
- 同网格存在锚点时，锚点优先
- 同格多普通站时，按“最接近最近锚点值”选值
- 硬锚点在平滑后保持不变
- 软观测点只允许受限调整

### 13.3 后端一致性验收

- CPU 与 GPU 同帧结果误差在允许范围内
- 图例着色分档一致
- 硬锚点、软观测格和雨区边界行为一致

### 13.4 性能验收

- 首帧生成时间达到目标量级
- 大时间范围批量计算不导致页面卡死
- 浏览器内存不随帧数无界增长

---

## 14. 开发建议顺序

建议按以下顺序落地：

1. 固化静态资产格式
2. 固化数据包协议
3. 先完成 CPU 基线算法
4. 用 CPU 结果作为标准答案
5. 再实现 GPU 后端
6. 完成主线程渲染与时间切换
7. 完成性能压测与参数校准

## 15. 当前仍需在开发阶段校准的参数

以下参数已具备方向，但仍建议用样例数据校准：

- `rain_mask` 扩展半径
- 连续传播裁决权重
- 平滑轮数
- 软观测格最大允许改变量
- 异常值判定阈值
- 深紫色最终色值

说明：

- 上述参数应做成可配置项
- 不应写死在算法核心逻辑中

## 16. 结论

本方案已满足以下核心约束：

- 计算在浏览器端完成，不占用服务器计算资源
- 下载完成后再计算，不受中途网络波动影响
- 有 GPU 时优先使用 GPU，无 GPU 时使用 CPU
- 输出同时满足连续色带等值面和网格雨量值
- 固定锚点与动态锚点共同主控
- 全部有效站点参与成图
- 零雨量站不参与计算，非雨区透明显示

按本文即可进入开发拆解阶段。
