## 2. 当前仍未收口事项

### 2.1 `rain_mask` 扩展半径当前仅为启发式默认实现

当前已实现规则：

- 分时产品默认半径范围：`3 ~ 10`
- 累计产品默认半径范围：`6 ~ 20`
- 当前半径估算采用：
  - `sqrt(knownGridCount)` 的密度项
  - 活动硬锚点最近邻间距换算得到的稀疏项
  - 二者取大后再做上下限夹逼

当前缺口：

- 这不是最终校准结果
- 仍需样例联调验证是否会出现局地雨区过宽或过窄
- 硬锚点额外扩展步数 `hardAnchorBonus = 1` 也仍属暂定参数

相关代码：

- `app/application/rain_iso/mask/estimate-expansion-radius.ts`
- `app/application/rain_iso/mask/build-rain-mask.ts`

结论：

- 这是当前 CPU 基线可运行所需的临时参数化实现，后续需在 `Task 20` 校准

### 2.2 受约束平滑参数当前仅为默认实现

当前已实现规则：

- 仅在 `rain_mask` 内平滑
- `hard_anchor_mask = 1` 的格点严格锁定
- `soft_obs_mask = 1` 的格点按原值上下阈值夹逼
- 普通补全格直接按局部均值更新

当前默认参数：

- `smooth_rounds = 2`
- `soft_obs_max_delta = 1.5`

当前缺口：

- 分时和累计产品是否应使用不同平滑轮数
- 软观测改变量阈值是否应回到正式参数表中的 `1.0`
- 当前均值平滑是否足够，是否需要更强的核或多轮策略

相关代码：

- `app/infrastructure/rain_iso/cpu/constrained-smooth.ts`
- `app/infrastructure/rain_iso/cpu/smooth-params.ts`

结论：

- 当前实现满足 CPU 基线闭环，但参数与核形式都仍待样例校准

### 2.3 Worker 批量执行协议当前仍是最小实现

当前已实现状态：

- `Task 14` 已打通 `load_assets -> start_task -> frame_ready -> task_completed/task_cancelled`
- 动态包会在 Worker 内先做完整双序列校验，通过后才逐帧启动
- 批量帧当前按 `frame_time` 升序调度
- 若同一时刻同时存在两类产品，当前稳定顺序为：
  - `rain_5m`
  - `accum_1h_step`

当前仍属占位或最小实现的部分：

- `task_progress.phase` 当前只回传粗粒度的 `assembling`
- 尚未把单帧链路细分为 `preprocessing / selecting_anchors / building_mask / smoothing` 等阶段事件
- `release_frame_cache` 当前为 no-op，占位到 `Task 18`
- Worker 当前仅支持 `CPU` 后端执行，`WebGPU / WebGL2` 仍待 `Task 15-16`

影响范围：

- 不阻塞当前浏览器批量 CPU 链路
- 但主线程若需要更细的阶段进度条或真实缓存释放行为，后续仍需补实现

相关代码：

- `app/interfaces/rain_iso/worker-entry.ts`
- `app/application/rain_iso/scheduler/frame-scheduler.ts`
- `app/application/rain_iso/scheduler/task-runner.ts`
- `app/interfaces/rain_iso/worker-client.ts`

结论：

- 这是“链路已闭环，但事件粒度和缓存接口仍属首版最小实现”的开放项

## 3. 当前不再视为未收口的事项

以下内容已经完成同步或冻结，不再作为偏差项保留：

1. 原始接口结构已按样例冻结
2. 动态输入模式已改为两份原始接口文件
3. 累计产品不再由前端从 `5分钟` 数据推导
4. 固定锚点来源已改为外部字典与 `station_meta` 求交集
5. 固定锚点统一视为同一优先级，后续不再细分类型
6. `5km` 邻域稀疏站已接入静态 `fallback_nearest_neighbors`
7. 邻域不足 `4` 站时仍保留动态锚点资格
8. 无坐标站点不纳入空间计算
9. 同格普通站完全等距时，先取值更大的站，再按 `stationId` 字典序稳定取第一条
10. 异常值业务例外当前不引入，不使用 `admauth / stlvl / star / isOut`
11. 若后续出现明确业务特判，再回到 `Task 6` 补充例外规则
12. 异常值基础判定规则已冻结并落地
