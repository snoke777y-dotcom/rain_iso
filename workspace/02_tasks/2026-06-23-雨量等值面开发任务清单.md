# 雨量等值面浏览器端开发任务清单

> **实施前置说明：**
> 1. 本清单基于最新设计文档：`workspace/01_prd/2026-06-23-雨量等值面浏览器端详细设计说明.md`
> 2. 当前仓库尚无现成前端工程骨架，本清单按“浏览器端 TypeScript + Worker + GPU/CPU 双后端”推荐落点拆分
> 3. 本版任务清单已按新需求重写，不再采用“仅融合站点主成图”的旧口径

**目标：** 落地浏览器端雨量等值面生成能力，输出 `5分钟分时等值面`、`起报累计等值面` 和对应 `1km x 1km` 网格雨量值。

**架构：** 以前端本地计算为核心，服务端仅提供静态资产和时间范围数据包。计算链路统一为“全站预处理 -> 固定锚点与动态锚点识别 -> 全站直接落格 -> 锚点优先裁决 -> 普通站同格择优 -> 连续传播补全 -> 硬锚点锁定平滑 -> 连续色带渲染”，执行后端按任务启动时预判为 GPU 或 CPU。

**建议技术栈：** TypeScript、Web Worker、WebGPU、WebGL2、Vitest、Playwright。

---

## 1. 任务拆分原则

- 先建立 `CPU 基线实现`，用来固化业务规则和作为 GPU 对照标准
- 再增加 `GPU 加速后端`
- 优先完成“能算对”，再优化“算得快”
- 先做“全站参与 + 分层约束”规则，再做性能优化
- 所有核心算法模块必须有独立单元测试
- 浏览器端流程必须有端到端接口测试
- 参数做成配置项，不在算法内部写死

## 3. 里程碑划分

### M1：CPU 基线可用

目标：

- 能读取静态资产和动态数据包
- 能生成 5 分钟分时和起报累计产品
- 能完成“锚点优先 + 全站参与”的单帧成图
- 能输出网格值和雨区掩膜

### M2：GPU 后端可用

目标：

- 可在任务启动前正确探测 GPU 能力
- 连续传播补全和平滑可在 GPU 上运行
- 与 CPU 结果误差满足设计要求

### M3：批量任务稳定

目标：

- 支持较长时间范围整包下载后批量生成
- 支持时间顺序逐张切换查看
- 性能、缓存、内存符合目标

## 7. 开发阶段必须重点盯住的风险

### 风险 1：全站参与后图面变脏

应对：

- 先完成同格择优与软观测约束
- 不允许普通站无约束地压过锚点

### 风险 2：无活动锚点时结果失稳

应对：

- 明确实现 `ordinary_only_mode`
- 为该模式单独写回归测试

### 风险 3：GPU 与 CPU 结果不一致

应对：

- 先做 CPU 真值
- 所有 GPU 结果都要对比 CPU 输出

### 风险 4：累计产品时间语义做错

应对：

- 单独对 `起报累计` 写回归测试
- 特别覆盖 `T0 非整点` 场景

### 风险 5：浏览器内存增长失控

应对：

- 中间数组复用
- 结果帧做 LRU
- 批量计算时及时释放中间纹理

## 8. 交付顺序建议

建议交付顺序：

1. `CPU 单帧正确`
2. `CPU 批量任务可跑通`
3. `页面可查看结果`
4. `GPU 加速接入`
5. `回归与样例校准`

这样做的原因是：

- 先证明业务规则正确
- 再证明浏览器工作流可行
- 最后再做性能拔高

## 9. 完成判定

以下条件同时满足，才可视为开发完成：

- 两类产品均可生成
- 页面可按时间顺序逐张查看
- 网格值可查询
- 0 值站点不参与计算
- `<0.1mm` 不渲染
- 非雨区透明
- 锚点优先于普通站
- 无活动锚点时 CPU 与 GPU 均可进入回退模式
- GPU 可用时优先用 GPU
- GPU 不可用时 CPU 可完整兜底
- 长时间范围任务不因内存问题失稳
- 核心业务规则均被测试覆盖
